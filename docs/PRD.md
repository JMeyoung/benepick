# 혜택 통합 추천 서비스 총정리

## 문서 개요
이 문서는 카드사, 통신사, 학생·연령 조건 기반 할인 혜택을 한 번에 모아 보여주는 서비스 아이디어를 사업 관점, 제품 관점, 유입 관점, 구현 관점에서 한 번에 정리한 MD 문서다. 초기 목표는 “모든 혜택을 완벽 자동 수집하는 플랫폼”이 아니라, 사용자가 자신의 조건에 맞는 혜택을 빠르게 찾을 수 있게 해주는 신뢰 가능한 웹 MVP를 만드는 데 있다.

## 1. 문제 정의
현재 할인 혜택 정보는 카드 비교 서비스, 통신사 멤버십 페이지, 학생 전용 할인 플랫폼 등으로 분산되어 있다. 카드고릴라는 1,500여 개 카드 비교를 제공하고, 뱅크샐러드는 224개 카드 비교 화면을 제공하며, KT 멤버십과 학생 할인 플랫폼인 UNiDAYS·Student Beans도 각각 별도의 채널에서 혜택 정보를 제공한다.[cite:10][cite:11][cite:20][cite:15][cite:18]

이 구조에서는 사용자가 “내가 지금 실제로 받을 수 있는 혜택”을 한 번에 보기 어렵다. 카드 혜택, 통신 혜택, 학생 할인 혜택이 서로 다른 서비스 안에 갇혀 있어 탐색 비용이 높고, 실제 절약 가능성을 바로 파악하기 어렵다.[cite:10][cite:11][cite:20][cite:15][cite:18]

## 2. 서비스 한 줄 정의
사용자가 나이, 학생 여부, 통신사, 카드사, 관심 업종을 입력하면 자신에게 적용 가능한 할인 혜택을 한 화면에서 정리해서 보여주는 혜택 통합 추천 웹서비스다.

## 3. 핵심 가치 제안
### 사용자 가치
- 여러 회사 사이트를 각각 방문하지 않아도 된다.
- “내 조건에서 쓸 수 있는 혜택”만 필터링해서 볼 수 있다.
- 카드, 통신, 학생 할인 정보를 같은 기준으로 비교할 수 있다.
- 원문 링크와 갱신일을 함께 보여 신뢰를 확보할 수 있다.

### 운영 가치
- 초기에는 공개된 공식 페이지를 사람이 검수해 등록하므로 정확도를 통제하기 쉽다.
- 데이터 구조를 먼저 잡아두면 추후 자동화, 제휴, 추천 고도화로 확장하기 쉽다.

## 4. 초기 타깃 사용자
초기 타깃은 학생 할인과 통신·생활 할인에 민감한 20대 대학생·청년, 그리고 교육·생활 정보 탐색이 잦은 학부모층으로 잡는 것이 적절하다. 실제로 강남엄마는 학원 정보를 지역·과목 기준으로 탐색하고 학부모 리뷰를 제공하는 구조를 갖고 있으며, 디스쿨도 교육카페와 지역카페 결합형 커뮤니티를 표방하고 있다.[cite:124][cite:128][cite:131]

초기 제품 검증은 너무 넓은 대상을 잡기보다 “생활밀착 할인 정보를 자주 찾는 사용자”에게 맞추는 편이 낫다. 학생 할인 플랫폼과 통신사 멤버십 페이지가 이미 별도 수요를 증명하고 있기 때문에, 이를 한 화면에 합쳐주는 경험이 초기 차별점이 된다.[cite:20][cite:15][cite:18]

## 5. MVP 범위
### 포함할 것
1. 나이대, 학생 여부, 통신사, 보유 카드사, 관심 카테고리 입력
2. 조건 기반 혜택 목록 조회
3. 혜택 상세 페이지
4. 저장 기능
5. 원문 링크, 갱신일, 종료 여부 표시
6. 관리자 혜택 등록/수정/종료 기능
7. 대기자 등록 및 인터뷰 신청 폼

### 제외할 것
- 카드 실적 자동 계산
- 카드 결제내역 연동
- 모든 혜택의 실시간 자동 크롤링
- 복수 혜택 중복 적용 최적화 계산
- 네이티브 앱 개발
- 복잡한 AI 추천 모델

## 6. 사용자 플로우
### 기본 플로우
1. 랜딩 페이지 유입
2. 조건 입력
3. 맞춤 혜택 목록 확인
4. 상세 혜택 확인
5. 저장 또는 원문 이동

### 운영 플로우
1. 운영자가 공식 출처 확인
2. 혜택 초안 등록
3. 조건 규칙 구조화 입력
4. 검수 후 공개
5. 종료 또는 내용 변경 시 수정

## 7. 정보 구조
### 사용자 화면
- 랜딩 페이지
- 온보딩/조건 입력
- 혜택 목록
- 혜택 상세
- 저장한 혜택
- 대기자 등록/인터뷰 신청

### 관리자 화면

- 혜택 목록 관리
- 혜택 등록/수정
- 상태 관리(draft/published/archived)
- 변경 이력 및 검수 메모

## 8. 데이터 구조 방향
이 서비스의 핵심은 혜택 설명을 텍스트로만 저장하지 않고, 조건을 구조화하는 것이다. 예를 들어 학생 여부, 통신사, 카드사, 연령 조건을 `eligibility_rules` 같은 테이블로 분리하면 필터링과 매칭 로직을 재사용하기 쉬워진다.

### 추천 핵심 테이블
| 테이블 | 설명 |
|---|---|
| users | 관리자 및 향후 일반회원 |
| user_profiles | 사용자 조건 정보 |
| organizations | 카드사, 통신사, 학생 플랫폼, 제휴사 |
| benefits | 혜택 기본 정보 |
| benefit_organizations | 혜택과 제공사/제휴사 연결 |
| eligibility_rules | 노출 조건 구조화 |
| favorites | 저장 기능 |
| lead_submissions | 광고 및 커뮤니티 유입 리드 |
| audit_logs | 관리자 작업 기록 |

## 9. 혜택 매칭 방식
초기 버전은 규칙 기반 필터링으로 충분하다. 학생 전용 혜택은 학생일 때만, 특정 통신사 혜택은 해당 통신사 사용자일 때만, 특정 카드사 혜택은 보유 카드 목록과 교집합이 있을 때만 노출하면 된다.

정렬은 점수 기반으로 단순화하면 된다.
- 관심 카테고리 일치 가산점
- 최근 업데이트 혜택 가산점
- 지역 일치 가산점
- 운영자가 지정한 추천 혜택 가산점

## 10. 데이터 수집 및 운영 원칙
초기에는 100% 자동 수집보다, 공식 공개 페이지를 기준으로 사람이 검수해 입력하는 구조가 적절하다. 카드고릴라, 뱅크샐러드, KT 멤버십, UNiDAYS, Student Beans처럼 혜택 카탈로그가 이미 각자 따로 운영되고 있으므로, 먼저 이를 정규화하는 운영 역량이 중요하다.[cite:10][cite:11][cite:20][cite:15][cite:18]

운영 원칙은 다음과 같다.
- 반드시 공식 출처 링크를 저장한다.
- 갱신일을 표시한다.
- 종료 가능성이 있는 혜택은 상태값으로 관리한다.
- 혜택 설명보다 조건 구조화를 우선한다.

## 11. 초기 유입 전략
초기 유입 채널은 인스타그램 광고와 교육·생활 커뮤니티를 병행하는 방식이 적합하다. 커뮤니티는 타깃 적합도가 높고, 인스타그램 광고는 빠르게 도달을 넓힐 수 있다는 장점이 있다.[cite:124][cite:128][cite:131][cite:144][cite:149]

### 왜 카페가 중요한가
강남엄마는 지역·학원·리뷰 중심 정보 구조를 제공하고, 디스쿨은 교육과 지역 커뮤니티 결합형 서비스를 표방하고 있어 교육·생활 정보에 민감한 사용자층과 맞닿아 있다.[cite:124][cite:128][cite:131] 따라서 아무 맘카페보다 교육 특화 커뮤니티를 먼저 탐색하는 편이 효율적이다.

### 우선 조사할 커뮤니티
| 우선순위 | 채널 | 이유 |
|---|---|---|
| 1 | 강남엄마 | 지역·학원·리뷰 중심이라 교육 생활 정보 수요가 뚜렷함 [cite:124][cite:128] |
| 2 | 디스쿨 | 교육+지역 커뮤니티 결합 구조 [cite:131] |
| 3 | 상위권 교육정보 커뮤니티 | 교육 정보 밀도가 높음 [cite:129][cite:132] |
| 4 | 대치동 학부모 카페 후보 | 지역 밀착형 정보 접근 가능성 [cite:111][cite:112] |

### 카페 글 게시 전 확인 항목
- 최근 게시글 활동량
- 상업성 글 허용 여부
- 설문·인터뷰 모집 허용 여부
- 링크 허용 여부
- 관리자 승인 필요 여부

## 12. 인스타그램 광고 전략
Meta는 Ads Manager에서 `Leads` 목표를 선택하면 `Instant forms`를 전환 위치로 사용할 수 있다고 설명한다.[cite:143] 또한 Meta for Business는 Facebook과 Instagram 광고 제작 및 관리 기능을 제공하고, Instagram 광고 역시 Meta 환경 안에서 운영할 수 있도록 안내한다.[cite:144][cite:149]

Lead Ads 모범 사례 문서는 인트로 섹션 사용, 양식 단순화, 사전 입력 질문 활용, 명확한 CTA 설정을 권장한다.[cite:94] Meta의 리드 생성 가이드는 웹사이트로 이동한 뒤에도 인앱 브라우저 안에서 즉시 양식을 붙이는 방식을 설명한다.[cite:95]

### 추천 광고 퍼널
1. 인스타그램 광고 노출
2. 즉시 양식 또는 서비스 내 리드 폼 제출
3. 대기자 등록 또는 인터뷰 신청 접수
4. 후속 연락 또는 베타 안내 전송

### 광고 문구 방향
- 문제 제기형: “혜택 찾느라 여러 사이트를 돌고 있지 않나요?”
- 효율형: “내 통신사·카드·학생 조건에 맞는 할인만 한 번에 확인”
- 검증형: “베타 테스트 참여자 모집 중”

### 폼 필수 요소
- 최소한의 질문
- 개인정보 수집 동의
- 개인정보처리방침 URL
- 완료 후 다음 행동 안내

## 13. 기술 스택 제안
### 권장 스택
| 영역 | 제안 |
|---|---|
| 프론트엔드 | Next.js App Router + TypeScript |
| UI | Tailwind CSS + shadcn/ui |
| 백엔드 | Next.js Route Handlers / Server Actions |
| DB | PostgreSQL |
| ORM | Prisma |
| 인증 | Clerk 또는 Auth.js |
| 배포 | Vercel |
| 분석 | PostHog 또는 GA4 |

이 구조는 초기 MVP에서 프론트와 백엔드를 한 리포지토리로 운영하기 좋고, 관리자와 사용자 화면을 빠르게 함께 개발할 수 있다는 장점이 있다. Claude Code는 코드베이스를 이해하고 파일을 수정하며 명령어를 실행하는 터미널 기반 에이전트형 코딩 도구로 소개되므로, 단일 리포지토리 구조와 기능 단위 작업 분해가 특히 잘 맞는다.[cite:140][cite:141]

## 14. Claude Code로 구현하는 방식
Anthropic은 Claude 관련 개발 가이드와 Claude Code 문서를 제공하고 있으며, Claude Code는 터미널·IDE 환경에서 코드베이스를 이해하고 수정하는 에이전트형 도구로 설명된다.[cite:138][cite:140][cite:141] 따라서 구현은 “한 번에 전체를 만들어 달라”보다 “작은 기능 단위로 쪼개서 검증하며 진행”하는 방식이 적합하다.

### 권장 개발 순서
1. 프로젝트 부트스트랩
2. 디자인 시스템과 공통 레이아웃 구축
3. Prisma 스키마와 마이그레이션 작성
4. 랜딩/온보딩/혜택 리스트 read-only 구현
5. 관리자 CRUD 구현
6. 저장 기능과 이벤트 로깅 추가
7. 대기자 등록 및 인터뷰 신청 폼 추가
8. 테스트와 배포 자동화

### Claude Code 사용 원칙
- 한 세션에는 하나의 큰 기능만 맡긴다.
- 요구사항과 완료 조건을 명확히 적는다.
- 테스트 명령과 기대 결과를 함께 요청한다.
- DB, API, UI 변경을 한 번에 과도하게 섞지 않는다.
- 작은 PR 단위로 나눈다.

| 유입 품질 저하 | 광고 리드의 품질이 낮을 수 있음 | 질문 수 최소화, 채널 태깅, 후속 검수 |
| 운영 부담 | 초기 수작업 입력이 많음 | 카테고리 우선순위 제한 |
| 범위 과대화 | 비교 서비스 전체를 한 번에 하려는 유혹 | MVP 범위 고정 |

## 18. Definition of Done
다음 조건을 만족하면 1차 MVP 완료로 본다.
- 사용자가 조건 입력 후 맞춤 혜택을 조회할 수 있다.
- 운영자가 관리자에서 혜택을 등록·수정·종료할 수 있다.
- 모든 혜택에 출처 링크와 갱신일이 있다.
- 광고/커뮤니티 유입용 리드 폼이 있다.
- UTM과 기본 이벤트 로그가 저장된다.
- 주요 사용자 플로우 테스트가 통과한다.

## 19. 최종 정리
이 프로젝트의 본질은 “혜택 정보 수집”이 아니라 “사용자가 자기 조건에서 바로 쓸 수 있는 혜택을 빠르게 발견하게 만드는 경험”이다. 따라서 초기 우선순위는 자동화보다 데이터 구조화, 운영 가능성, 조건 기반 노출, 사용자 입력 경험, 유입 추적에 두는 것이 맞다.

초기 채널은 교육·생활 커뮤니티와 인스타그램 광고를 병행하고, 제품은 웹 MVP로 시작해 운영 검증을 거친 뒤 자동화와 앱 확장을 검토하는 방식이 가장 현실적이다.[cite:124][cite:128][cite:131][cite:143][cite:144][cite:149] 구현은 Claude Code의 강점을 살려 작은 작업 단위, 명확한 완료 조건, 반복 테스트 중심으로 진행하는 것이 적절하다.[cite:138][cite:140][cite:141]
